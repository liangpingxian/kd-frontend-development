## 第一章：概述

### 1.1 什么是脚本控制器

脚本控制器（Script Controller）是基于 KingScript 的 Web API 控制器，运行在 BOS 平台的 KWC（Kingdee Web Controller）框架中。它允许开发者使用脚本语言快速创建 RESTful API，无需编译 Java 代码。

**核心特性：**
- 🚀 **快速开发**：使用 KingScript 编写，即时生效
- 🔧 **灵活配置**：通过 .kws 元数据配置 URL 路由和方法绑定
- 🛡️ **权限控制**：内置权限验证机制
- 📊 **数据库集成**：可访问业务数据实体
- 🔄 **热部署**：支持运行时更新，无需重启服务

### 1.2 应用场景

✅ **适合使用脚本控制器的场景：**
- 快速原型开发和 PoC 验证
- 简单的 CRUD 接口
- 第三方系统集成接口
- 临时性业务逻辑处理
- 需要频繁修改的业务规则

❌ **不适合使用脚本控制器的场景：**
- 复杂的业务逻辑（建议使用 Java 控制器）
- 高性能要求的接口
- 需要事务控制的场景
- 涉及底层系统调用的功能

## 第二章：快速开始

### 2.1 第一个脚本控制器

#### 步骤 1：创建控制器配置

创建 `UserController.kws` 文件：

```xml
<?xml version="1.0" encoding="UTF-8"?>
<Controller>
    <name>UserScriptController</name>
    <isv>kingdee</isv>
    <app>dev</app>
    <version>1</version>
    <url>/kd/dev/sample/users</url>
    <scriptFile>UserScriptController.ts</scriptFile>
    <methods>
        <method>
            <name>getUser</name>
            <url>/{id}</url>
            <httpMethod>GET</httpMethod>
            <permission>
                <permission>
                    <permitAll>false</permitAll>
                    <entityNumber>bos_user</entityNumber>
                    <permItemId>47150e89000000ac</permItemId>
                    <checkRightApp>dev</checkRightApp>
                </permission>
            </permission>
        </method>
    </methods>
</Controller>
```

#### 步骤 2：编写脚本代码

创建 `UserScriptController.ts` 文件：

```typescript
class UserScriptController {
  getUser(request: any, response: any) {
    // 获取路径参数
    const userId = request.getLongPathVariable('id');

    // 返回用户信息
    const userData = {
      id: userId,
      name: '张三',
      email: 'zhangsan@example.com'
    };

    response.ok(userData);
  }
}

let kwcController = new UserScriptController();
export { kwcController };
```

#### 步骤 3：部署控制器

通过脚手架命令上传控制器：

```bash
# 使用脚手架命令部署控制器
bos-cli deploy --config UserController.kws --script UserScriptController.ts
```

脚手架会自动完成以下操作：
- 读取 .kws 元数据文件
- 打包脚本文件
- 上传到服务器
- 注册控制器路由

#### 步骤 4：测试接口

访问接口：
```
GET ../kwc/v1/kd/dev/sample/users/123
```

返回结果（HTTP 200）：
```json
{
  "id": 123,
  "name": "张三",
  "email": "zhangsan@example.com"
}
```

## 第三章：控制器配置详解

### 3.1 必填字段清单

控制器配置必须包含以下字段，否则部署会失败：

| 字段 | 说明 | 示例 | 是否必填 |
|------|------|------|---------|
| `name` | 控制器名称（唯一标识） | `UserScriptController` | ✅ 必填 |
| `isv` | 开发商编码 | `kingdee` 或自定义编码 | ✅ 必填 |
| `app` | 业务应用编码 | `dev`、`bos` 等 | ✅ 必填 |
| `version` | 版本号（正整数） | `1`、`2`、`3` | ✅ 必填 |
| `url` | 控制器根 URL 地址 | `/kd/dev/sample/users` | ✅ 必填 |
| `scriptFile` | 脚本文件名 | `UserScriptController.ts` | ✅ 必填 |
| `methods` | 方法定义集合（至少 1 个） | 见 3.2 节 | ✅ 必填 |

### 3.2 .kws 元数据模板

```xml
<?xml version="1.0" encoding="UTF-8"?>
<Controller>
    <!-- 基础信息 -->
    <name>UserScriptController</name>
    <isv>kingdee</isv>
    <app>dev</app>
    <version>1</version>

    <!-- URL 地址 -->
    <url>/kd/dev/sample/users</url>

    <!-- 脚本文件绑定 -->
    <scriptFile>UserScriptController.ts</scriptFile>

    <!-- 方法定义 -->
    <methods>
        <method>
            <name>getUser</name>
            <url>/{id}</url>
            <httpMethod>GET</httpMethod>

            <!-- 权限配置（必填） -->
            <permission>
                <permission>
                    <permitAll>false</permitAll>
                    <anonymousUser>false</anonymousUser>
                    <entityNumber>bos_user</entityNumber>
                    <permItemId>47150e89000000ac</permItemId>
                    <checkRightApp>dev</checkRightApp>
                </permission>
            </permission>
        </method>

        <method>
            <name>createUser</name>
            <url></url>
            <httpMethod>POST</httpMethod>
            <permission>
                <permission>
                    <permitAll>false</permitAll>
                    <anonymousUser>false</anonymousUser>
                    <entityNumber>bos_user</entityNumber>
                    <permItemId>47150e89000000ac</permItemId>
                    <checkRightApp>dev</checkRightApp>
                </permission>
            </permission>
        </method>
    </methods>
</Controller>
```

### 3.3 URL 地址规则（重要）

#### 3.3.1 URL 格式要求

```
/{开发商}/{应用标识}[/自定义子目录]/{资源复数}
```

**示例：**
- ✅ `/kd/dev/sample/users` （推荐）
- ✅ `/kd/bos/usercenter/users` （推荐）
- ✅ `/myisv/myapp/orders` （自定义开发商）
- ❌ `/kd/dev` （缺少资源路径，至少 3 级）
- ❌ `/dev/sample/users` （缺少开发商前缀）

#### 3.3.2 开发商前缀规则

| isv 值 | URL 前缀 | 说明 |
|--------|---------|------|
| `kingdee` | `/kd/` | 金蝶原厂统一使用 `kd` |
| 其他值 | `/{isv}/` | 二开厂商使用自己的编码 |

#### 3.3.3 完整 URL 拼接规则

最终访问 URL = **类 URL** + **方法 URL**

| 类 URL | 方法 URL | 最终访问 URL |
|--------|---------|-------------|
| `/kd/dev/users` | `/{id}` | `/kd/dev/users/{id}` |
| `/kd/dev/users` | `` (空) | `/kd/dev/users` |
| `/kd/dev/users` | `/profile` | `/kd/dev/users/profile` |

**注意事项：**
- 方法 URL 为空时，直接使用类 URL
- 方法 URL 不以 `/` 开头会自动添加
- 类 URL 以 `/` 结尾会自动去重

### 3.4 方法配置（Method）

#### 3.4.1 必填字段

| 字段 | 说明 | 示例 | 是否必填 |
|------|------|------|---------|
| `name` | 方法名（对应脚本中的方法） | `getUser`、`createUser` | ✅ 必填 |
| `httpMethod` | HTTP 请求方法 | `GET`、`POST`、`PUT`、`DELETE` | ✅ 必填 |
| `permission` | 权限配置对象 | 见 3.4.2 | ✅ 必填 |

#### 3.4.2 权限配置

每个方法必须配置权限检查：

```xml
            <permission>
                <permission>
                    <permitAll>false</permitAll>
                    <anonymousUser>false</anonymousUser>
                    <entityNumber>bos_user</entityNumber>
                    <permItemId>47150e89000000ac</permItemId>
                    <checkRightApp>dev</checkRightApp>
                </permission>
            </permission>
```

**字段说明：**

| 字段 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `permitAll` | boolean | `false` | 是否不受权限控制。设为 `true` 则跳过统一的权限检查，由控制器方法自行处理权限逻辑 |
| `anonymousUser` | boolean | `false` | 是否允许匿名用户访问。需同时将 `permitAll` 设为 `true`，匿名访问才会生效 |
| `entityNumber` | string | — | 验权使用的业务实体编码（如 `bos_user`） |
| `permItemId` | string | — | 验权使用的权限项 ID（在权限设计中定义） |
| `checkRightApp` | string | — | 验权使用的业务应用编码（通常与 `app` 一致） |

**常见配置场景：**

```xml
<!-- 场景 1：标准权限验证（推荐） -->
<permission>
    <permission>
        <permitAll>false</permitAll>
        <entityNumber>bos_user</entityNumber>
        <permItemId>47150e89000000ac</permItemId>
        <checkRightApp>dev</checkRightApp>
    </permission>
</permission>

<!-- 场景 2：跳过统一权限检查，由控制器方法自行鉴权 -->
<permission>
    <permission>
        <permitAll>true</permitAll>
    </permission>
</permission>

<!-- 场景 3：允许匿名用户访问（需同时开启 permitAll） -->
<permission>
    <permission>
        <permitAll>true</permitAll>
        <anonymousUser>true</anonymousUser>
    </permission>
</permission>
```

### 3.5 版本管理规则

#### 3.5.1 版本号规则

- 类型：**正整数**（1, 2, 3...）
- 比较：新版本号必须 **大于** 已有版本号
- 覆盖：不支持相同版本号覆盖

#### 3.5.2 版本升级示例

```
<!-- 首次部署 -->
<version>1</version>

<!-- 升级部署 -->
<version>2</version>  <!-- ✅ 成功 -->

<!-- 错误示例 -->
<version>2</version>  <!-- ❌ 失败：版本相同 -->
<version>1</version>  <!-- ❌ 失败：版本更低 -->
```

### 3.6 部署验证流程

部署时会按以下顺序验证：

```
1. 必填字段检查
   ├─ name 是否为空
   ├─ isv 是否为空
   ├─ app 是否为空
   ├─ url 是否为空
   ├─ scriptFile 是否为空
   └─ methods 是否为空

2. URL 格式检查
   ├─ 是否包含开发商前缀
   ├─ 是否包含应用编码
   └─ 是否超过 2 级路径（必须有资源路径）

3. 方法配置检查
   ├─ method.name 是否为空
   ├─ method.httpMethod 是否为空
   └─ method.permission 是否配置

4. 版本号检查
   └─ 新版本号 > 已有版本号
```

## 第四章：脚本开发基础

### 4.1 脚本文件结构

#### 基本结构

```typescript
// 1. 定义控制器类
class MyController {
  // 2. 定义方法
  methodName(request: any, response: any) {
    // 业务逻辑
  }
}

// 3. 创建实例并导出
let kwcController = new MyController();
export { kwcController };
```

#### 命名规范

| 项目 | 规范 | 示例 |
|------|------|------|
| 类名 | PascalCase（大驼峰） | `UserScriptController` |
| 方法名 | camelCase（小驼峰） | `getUser`、`createOrder` |
| 文件名 | 与类名一致 | `UserScriptController.ts` |
| 完整类名 | `@isv/app/name/ScriptFile.ts` | `@kingdee/dev/UserScriptController.ts` |

### 4.2 KingScript 语法规范

KingScript 是基于 TypeScript 的增强版本，提供了更强大的类型系统和运行时优化。

```typescript
// KingScript - 标准写法 ✅
class MyController {
  getUser(request: any, response: any) {
    const userId: number = request.getLongPathVariable('id');
    response.ok({ id: userId });
  }
}
```

**KingScript 特性：**
- 强类型检查
- 自动导入系统类型
- 优化的运行时性能
- 更好的错误提示

### 4.3 多方法示例


## 第五章：请求处理 API

### 5.1 HTTP 方法获取

```typescript
const method = request.getHttpMethod();
// 返回："GET", "POST", "PUT", "DELETE" 等
```

### 5.2 路径参数获取

路径参数是从 URL 中提取的变量，例如 `/users/{id}` 中的 `id`。

```typescript
// 获取字符串类型
const idStr = request.getPathVariable('id');

// 获取长整型（推荐）
const userId = request.getLongPathVariable('id');
```

**示例：**
```typescript
// URL: /kd/dev/users/123
const userId = request.getLongPathVariable('id');
// userId = 123 (Long 类型)
```

### 5.3 查询参数获取

查询参数是 URL 问号后面的参数，例如 `/users?name=张三&age=25`。

```typescript
// 获取所有查询参数
const params = request.getQueryParams();

// 获取指定参数
const nameObj = request.getQueryParam('name');

// 获取字符串类型
const name = request.getStringQueryParam('name');

// 获取整型
const age = request.getIntQueryParam('age');

// 获取长整型
const longId = request.getLongQueryParam('id');

// 获取布尔值
const isActive = request.getBooleanQueryParam('active');

// 获取日期类型
const createDate = request.getDateQueryParam('createDate');
```

**示例：**
```kingscript
// URL: /users?name=张三&age=25&active=true
const name = request.getStringQueryParam('name');      // "张三"
const age = request.getIntQueryParam('age');           // 25
const isActive = request.getBooleanQueryParam('active'); // true
```

### 5.4 请求头处理

```typescript
// 获取所有请求头
const headers = request.getHeaders();

// 获取指定请求头
const authHeader = request.getHeader('Authorization');
const contentType = request.getHeader('Content-Type');
```

**示例：**
```kingscript
const token = request.getHeader('Authorization');
// 返回："Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9..."
```

### 5.5 请求体处理

```typescript
// 获取原始请求体（Object 类型）
const bodyObj = request.getBody();

// 获取字符串类型的请求体
const bodyStr = request.getStringBody();

// 获取 Map 类型的请求体（推荐）
const bodyMap = request.getMapBody();
```

**示例：**
```kingscript
// POST /users
// Body: {"username": "admin", "password": "123456"}

const body = request.getMapBody();
const username = body['username'];  // "admin"
const password = body['password'];  // "123456"
```

### 5.6 综合示例

```typescript
class UserController {
  // 综合使用各种参数
  processUser(request: any, response: any) {
    // 路径参数
    const userId = request.getLongPathVariable('id');

    // 查询参数
    const includeDetails = request.getBooleanQueryParam('includeDetails');
    const lang = request.getStringQueryParam('lang');

    // 请求头
    const token = request.getHeader('Authorization');

    // 请求体
    const body = request.getMapBody();
    const action = body['action'];

    // 业务逻辑处理
    const result = {
      userId: userId,
      includeDetails: includeDetails,
      lang: lang,
      token: token,
      action: action
    };

    response.ok(result);
  }
}
```

## 第六章：响应处理 API

### 6.1 成功响应

```typescript
// 返回简单数据
response.ok('操作成功');

// 返回数字
response.ok(100);

// 返回对象
response.ok({
  id: 123,
  name: '张三'
});

// 返回数组
response.ok([
  { id: 1, name: '用户 1' },
  { id: 2, name: '用户 2' }
]);
```

### 6.2 错误响应

使用 `throwException` 方法抛出业务异常：

```typescript
// 简单错误（使用 400 状态码）
response.throwException('操作失败', 400, 'OPERATION_FAILED');

// 带详细信息和业务错误码
response.throwException('参数无效：用户名不能为空', 400, 'INVALID_PARAM');

// 未授权错误
response.throwException('未登录或登录已过期', 401, 'UNAUTHORIZED');

// 服务器内部错误
response.throwException('系统繁忙，请稍后再试', 500, 'SYSTEM_ERROR');
```

**参数说明：**
- `message`: 异常提示消息
- `httpStatusCode`: HTTP 状态码（如 400=请求有误，401=未授权，500=服务器错误）
- `businessErrorCode`: 业务异常码（由业务自定义的错误编码）

**常见 HTTP 状态码：**
- `400` - Bad Request（请求参数错误）
- `401` - Unauthorized（未授权）
- `403` - Forbidden（禁止访问）
- `404` - Not Found（资源不存在）
- `500` - Internal Server Error（服务器内部错误）

### 6.3 设置 HTTP 状态码

使用 `of` 方法可以指定 HTTP 状态码：

```typescript
// 创建成功（201）
response.of(201, { message: '创建成功', id: 123 });

// 删除成功（204 No Content）
response.of(204, null);

// 请求有误（400）
response.of(400, { error: '参数格式不正确' });

// 未授权（401）
response.of(401, { error: '请先登录' });

// 资源不存在（404）
response.of(404, { error: '用户不存在' });
```

**注意：**
- `ok(data)` 方法等价于 `of(200, data)`
- 对于异常情况，建议使用 `throwException` 方法而不是 `of` 方法

### 6.4 响应数据格式

响应遵循 RESTful 风格，直接返回业务数据，使用 HTTP 状态码表示请求结果。

**成功响应（HTTP 2xx）：**
```json
{
  "id": 123,
  "name": "张三"
}
```

**错误响应（HTTP 4xx / 5xx）：**
```json
{
  "message": "参数无效：用户名不能为空",
  "code": "INVALID_PARAM"
}
```

**不同方法的响应示例：**

```typescript
// 示例 1：使用 ok 方法（HTTP 200）
response.ok({ id: 1, name: '测试' });
// HTTP 200，响应体：{"id":1,"name":"测试"}

// 示例 2：使用 of 方法（HTTP 201）
response.of(201, { message: '创建成功' });
// HTTP 201，响应体：{"message":"创建成功"}

// 示例 3：抛出异常（HTTP 400）
response.throwException('用户名已存在', 400, 'USER_EXISTS');
// HTTP 400，响应体：{"message":"用户名已存在","code":"USER_EXISTS"}
```

### 6.5 响应示例

```typescript
class ResponseExample {
  // 示例 1：标准成功响应
  successExample(request: any, response: any) {
    const data = {
      id: 1,
      name: '测试数据'
    };
    response.ok(data);
    // HTTP 200，响应体：{"id":1,"name":"测试数据"}
  }

  // 示例 2：带状态码的响应（创建成功）
  createdExample(request: any, response: any) {
    response.of(201, { message: '创建成功', id: 123 });
  }

  // 示例 3：错误处理
  errorExample(request: any, response: any) {
    try {
      // 业务逻辑
      throw new Error('业务异常');
    } catch (e) {
      response.throwException(e.getMessage(), 500, 'BUSINESS_ERROR');
    }
  }

  // 示例 4：参数验证失败
  validationExample(request: any, response: any) {
    const body = request.getMapBody();

    if (!body['username']) {
      response.throwException('用户名不能为空', 400, 'MISSING_USERNAME');
      return;
    }

    if (!body['email'] || !body['email'].includes('@')) {
      response.throwException('邮箱格式不正确', 400, 'INVALID_EMAIL');
      return;
    }

    // 验证通过，继续处理
    response.ok({ message: '验证通过' });
  }
}
```
